# AI Native 组织形式 × 指令驱动 Skill 进化

> 一份写给同事看的解读：把 Bttr 在 *Burn Tokens, Not Headcount* 中提出的"组织五层循环"，跟我们正在搭的"指令驱动 Skill 进化引擎"咬合在一起，看清楚两件事的同一性 —— **公司是一组会自我改进的循环；让循环越跑越聪明的钥匙，是把人嘴里那一句"该这样改"完整无损地存下来。**

> 出处声明：本文第 1 章核心观点与引文均来自 Bttr 团队公开文章 *Burn Tokens, Not Headcount* （[https://www.makebttr.com/insights/burn-tokens-not-headcount](https://www.makebttr.com/insights/burn-tokens-not-headcount)）。第 2 章引用我们内部《指令驱动 Skill 进化 · MVP 设计方案》文档原文，未做二次发挥。

## 第 1 章 · 深度解读 Bttr 五层循环

### 1.0 立论：公司不是组织架构图，是一组递归的自我改进循环

Bttr 这篇文章的开篇就把传统组织形式的合法性来源拆穿了：**公司今天看起来像罗马军团（Roman legion）那样的层级结构，并不是因为这种形态最优，而是因为人类在低带宽人际沟通时代不得不发明的"应对机制"（coping mechanism）**。它有用，是因为信息传递贵；而 AI 让信息传递的成本骤降到接近零，于是过去那个"必须要中层来协调"的前提消失了。

Bttr 给出的替代叙述很激进，但很干净：

> "A company is not an org chart. It is a set of recursive, self-improving loops."
> （公司不是一张组织架构图，它是一组递归的、自我改进的循环。）

这句话里有三个词值得拆开看：

- **loop（循环）**：公司里发生的每一件事 —— 接到客户工单、研发新产品、谈一笔合同、做一份招股书研报 —— 都不是一次性的"动作"，而是一组被反复执行的"流程"。流程意味着：你能观察它、能优化它、能让它下次跑得更好。
- **recursive（递归）**：循环的输出会变成下一次循环的输入。今天解决的一个 bug、做的一份模板、回的一封邮件，都会沉淀为下一次循环可以站在上面的"地基"。
- **self-improving（自我改进）**：递归的循环如果只是机械重复，那只是在原地打转；只有循环本身能被修改，"做事的方式"会因为做事的过程而变得更好，组织才会变聪明。

把"公司"这个名词换成"循环"这个动词，就解锁了一种新的看待组织的方式 —— **你不再问"我们应该有几个部门、多少层级"，你开始问"我们有几个核心循环、每个循环的五层都搭好了吗、它每跑一轮变得更聪明了吗"。**

接下来的五层（再加上一个环绕中心），就是 Bttr 给"一个能自我改进的循环"下的最小工程定义。

---

### 1.1 第一层 · Sensor Layer（感知层）

> "The loop only knows what it can sense."
> （循环只知道它能感知到的事物。）

**这一层是什么**：循环的输入感官 —— 客户邮件、支持工单、代码 commit、产品遥测、内部 Slack 消息、销售电话录音、会议记录、监控告警。一切"外部世界发生了什么"的信号，都要先穿过感知层进入循环。

**它在解决什么问题**：决定循环的"视野边界"。Bttr 把这一句话写得非常硬 —— 循环只能作用于它能感知到的事物。一件没有被传感器接住的事，对系统来说**就是没有发生过**。这听起来像废话，但它直接推翻了大量传统组织对"信息流"的设想：以前一件事被员工口头传达三层之后丢失，没人会觉得是个 bug；在循环视角下，这是感知层漏接的事故。

**举例**：YC 的做法被 Bttr 单独拎出来当典型 —— **2000 小时合伙人录音**全都喂进系统，让 AI 在一个周末重生成 150 页用户手册，从此每月自动更新。这就是把"老合伙人脑子里的隐性知识"变成感知层的常规输入。再举一个更日常的：你和分析师的每一次微信对话、每一条修改意见，如果没有被 log 到一个 agent 看得到的地方，那这条信息就根本没有"发生过"。我们做指令驱动 Skill 进化的第一个动作 —— "记录指令" —— 本质上就是在补这一层的传感器。

**反思**：Sensor 层最容易被低估，因为搭起来枯燥，写不出炫的 demo。但**循环的天花板就是它的感知力**。一个看不见自己客户在抱怨什么的循环，再聪明的策略层也救不回来。

---

### 1.2 第二层 · Policy Layer（策略层）

> "Rules about what the system is allowed to do without asking."
> （关于系统在不询问的情况下可以做什么的规则。）

**这一层是什么**：定义"什么动作系统可以自己拍板、什么必须升级到人"的规则。它不是某个 if/else 块，而是一套**护栏（guardrails）**。

**它在解决什么问题**：自治与监督的分界。这一层让你回答下面这种问题：

- agent 可以自己改一段措辞吗？可以。
- agent 可以自己往招股书里补一段判断吗？可以，但要标注是 AI 推断。
- agent 可以自己把一个数字从财报里改掉吗？不可以，必须升级。
- agent 可以自己给客户回复退款邮件吗？低于 ¥500 可以，超过要升级。

策略层的精髓是**默认值**：它把"哪些事不需要每次问人"显式地写下来，于是循环跑起来才不会每一步都卡在审批上。同时它也把"哪些事必须问人"显式地写下来，于是循环跑起来不会闯祸。

**举例**：在 SpaceX 招股书研究循环里，我们的策略层会规定：agent 可以自己重组段落、可以自己加交叉引用、可以自己对比同业；但任何涉及"我对这家公司估值的最终判断"，必须由分析师做出 —— 这是一个角色边界。Bttr 的金句是："Senior people will increasingly act like fund partners. Outcome over output."（高级人才会越来越像基金合伙人，要的是结果不是产出。）策略层就是把这种角色边界落到代码里。

**反思**：策略层是 prompt 之外的一道闸门。Bttr 一句话点破了它的位置：

> "The line between autonomy and oversight is set here, not at the prompt."

很多人把 agent 的"敢做什么、不敢做什么"写在 prompt 里，但 prompt 是软约束，模型可以被绕过；策略层是工程化的硬约束，挂在循环外侧，agent 根本到不了那一步。

---

### 1.3 第三层 · Tool Layer（工具层）

> "Deterministic APIs ... Boring infrastructure."
> （确定性 API ……无聊的基础设施。）

**这一层是什么**：循环里所有"动手做事"的能力，被暴露成一组**确定性 API** —— 查数据库、读日历、发发票、抓网页、调财报接口、跑 SQL、改文件。注意"确定性"三个字：同样的输入要给出同样的输出，行为可预测、可测试、可审计。

**它在解决什么问题**：把循环的"能力面"和"决策面"分开。决策面是 agent 自己用语言模型推理出来的，工具面则是死板的、机械的、可信的。循环依赖的不是"模型很聪明所以会做对"，而是"模型再不聪明，调用 send_invoice() 也只能按既定逻辑发出一张合规发票"。

**举例**：YC 那个老司机案例里，**第二个 agent 自动 review 第一个 agent 提交的 MR、合并、部署**。这两个 agent 自己的脑子都不太可能保证次次靠谱，但**部署这一步是死板的工具**：通过 review 就部署，没通过就不部署。这就是工具层在干活。落到我们的项目，"读取 spacex_s1_clean.txt 中的某一段"应该是一个工具，"写入 instructions 表"应该是一个工具，"调反思器" 应该是一个工具 —— 而不是 agent 用魔法做完。

**反思**：Bttr 用的形容词是 "boring"。听起来贬义，其实是高度褒义 —— **工具层越无聊越好**。无聊意味着确定性、意味着工程化、意味着调试方便。一个公司是不是 AI Native，看它的工具层有多厚就知道：薄的公司一切靠 agent 临场发挥；厚的公司 agent 只负责选用哪一个工具。

---

### 1.4 第四层 · Quality Gate（质量闸门）

> "Safety filters and human review for high-stakes actions."

**这一层是什么**：所有从循环里被推出去的产出 —— 一段写给客户的回复、一份生成的报告、一行代码 patch、一笔退款 —— 在送达世界之前要经过的一道闸门。质量闸门由两部分组成：**安全过滤器（自动）+ 高风险动作的人工审核**。

**它在解决什么问题**：把"自治"和"监督"用看得见的方式落地。Policy 层说"哪些事可以自己做"，是规则；Quality Gate 是这条规则的**执行点** —— 闸门在这里立着，任何要出去的东西都要在这里被检查一道。

**举例**：在我们的 Skill 进化引擎里，这一层就是**留出集 + LLM-as-judge 打分 + 人工最终确认**。MVP 文档原话是："新版得分 ≥ 基线才进入候选；否则丢弃……通过验证的候选，由你一键确认上线。skill 变更是高风险动作，人在闭环。" 这就是一个标准的 Quality Gate：先自动过一道（留出集打分），再人工拍板（一键确认）。

**反思**：Quality Gate 的存在让"自治"变得可负担。**没有闸门的自治，要么不敢放开（agent 永远只能写草稿）、要么放开就出事（agent 直接发了一封不该发的邮件）**。有了闸门，才能放心让循环去跑、出错也只会出在闸门内侧。Bttr 的原句很硬：

> "The line between autonomy and oversight is set here, not at the prompt."

这句话出现在策略层和质量闸门之间反复回响 —— 它在提醒你：**自治程度不是用嘴说"agent 可以放手做"决定的，是用闸门设在哪里决定的。**

---

### 1.5 第五层 · Learning Mechanism（学习机制）

> "The system looks at where it failed, writes the fix, ships it. By morning it knows more than it did last night."
> （系统看着自己在哪里失败了，写出补丁，部署上线。第二天早晨，它比昨晚知道得更多。）

**这一层是什么**：循环的"自检 + 自改 + 自部署"能力。失败被检测到 → 诊断写出来 → 补丁产生 → 通过质量闸门 → 上线。而且这件事在循环本身的运转中自然发生，不需要专门拉一个工程师来排期。

**它在解决什么问题**：让系统结构性地变聪明，而不只是变快。Bttr 把这一层称作"**多数公司缺的那一层**"（the missing layer in most companies）。前四层很多公司都有 —— 数据采进来（Sensor）、规则写好（Policy）、API 暴露（Tool）、QA 流程（Gate）—— 但跑了一年之后还是去年那个公司，因为没有第五层。

**举例**：Bttr 给出的 YC 案例非常生动 —— 一次查询失败了，监控 agent 自动追问"为什么"：是错的工具？是缺的 skills 文件？是过期索引？是错误视图？诊断出来后，agent 写补丁、提 MR；第二个 agent review、合并、部署。**第二天早晨，整个 codebase 都比昨晚聪明了一点。**

> "Runs overnight. By morning the codebase is smarter."

落到我们 Skill 进化引擎，这一层就是**反思器**：把这一周积累的修改指令读进来，归纳出反复出现的模式，写出 skill 补丁，验证后上线。区别只在于 —— Bttr 的例子里被改的是代码，我们这里被改的是 skill。**核心动作完全同构：失败 → 反思 → 修复 → 部署 → 下一轮起点更高。**

**反思**：学习机制是把循环从"工具集合"变成"有机体"的关键。没有这一层，所有前四层的努力都只是把"今天的能力"工程化了；有了这一层，循环才会**让明天的能力比今天更强**。这是斜率（slope）问题，不是水平（level）问题。

---

### 1.6 中心 · Company Context（公司语境）

> "If it was not recorded, it did not happen to the intelligence."
> （没有被记录的东西，对智能体而言就是没有发生过。）

**这一层是什么**：环绕在五层循环中心的"公司大脑" —— 公司的历史、决策、客户、产品、流程、人员、教训、品味、行话、潜规则。所有让公司之所以成为这家公司的隐性知识，要被显式地、可读地放在一个地方，让循环里的每个 agent 都能读到。

**它在解决什么问题**：解决"这个 agent 不像我们公司的人"。一个通用 LLM 知道一切又什么都不知道；它知道波音和 SpaceX 是怎么回事，但不知道**你们公司**为什么决定不做 X 业务、为什么和某个客户绝交、为什么对某种文风过敏、为什么承付义务那一段必须拆期限分布。这些东西如果不写下来，agent 就不可能学会。

**举例**：YC 那 2000 小时合伙人录音 → 150 页用户手册 → 每月自动更新，本质上就是把合伙人脑子里的"YC 是什么"做成 Company Context，让所有 agent 在做事前可读。

**Bttr 的金句**：

> "Legibility is the unlock."
> （可读性，是一切的解锁开关。）

> "The company is a brain in the middle. Humans sit at the edge."
> （公司是中间一颗大脑，人坐在边缘。）

后一句是整篇文章里最颠覆传统组织观的一句。传统的组织观是"人在中间，公司架构服务于人的协作"；AI Native 的组织观是反过来的 —— **公司大脑在中间承载所有的循环和知识，人坐在边缘，做新情境下的判断、伦理、谈判、设计品味、创始人时刻**。中间那颗大脑越完整、越可读，边缘那个人越自由。

**反思**：Company Context 不是一层、是一个中心，因为它跟所有五层都打交道：

- Sensor 收来的信号要写进 Context；
- Policy 的规则本身存在 Context 里；
- Tool 要根据 Context 选用；
- Quality Gate 要参考 Context 来判断"这个产出像不像我们公司的"；
- Learning Mechanism 学到的新东西要回流到 Context。

可以这么理解：五层是循环的骨骼，Company Context 是循环的血液 —— 每一次循环跑动，都把新东西写回血液，下一次循环又把血液带到每一层。

---

### 1.7 五层 + 中心：合在一起看一次循环

把上面六个组件合在一起，一次循环大致长这样：

```
                ┌───── Sensor ─────┐
                │                  │
                ▼                  │
        Company Context  ◀────── Learning
       （会被读、会被写）           │
                │                  │
                ▼                  │
              Policy ───▶ Tool ──▶ Quality Gate ──▶ 输出到世界
```

每一次循环跑完，都会发生四件事：

1. Sensor 收到了新信号 → 写进 Context；
2. Policy 根据 Context 决定 agent 可以自己做、还是要升级；
3. Tool 在确定性 API 上动手做；
4. Quality Gate 把住门，安全的放出去，可疑的留下来；
5. Learning 把"留下来的可疑"诊断成补丁，下一次同类问题循环就更聪明。

> "If you were building your company today, would you start it in this shape? If you are small enough, you have no excuse."
> （如果你今天才开始建你的公司，你会以这种形态开始吗？如果你足够小，你没有借口。）

**这是 Bttr 这篇文章的钉子句。** 它没有让你立刻把公司翻新一遍，但它逼你回答一个问题：你今天做的每一件重复的事，能不能用这五层加一个中心的形态来重描？如果可以，就把它先描一下，再决定要不要改造。

## 第 2 章 · Skill 进化引擎四步动作

> 本章引用我们内部《指令驱动 Skill 进化 · MVP 设计方案》文档原文，原话尽量保留。

### 2.0 一句话方案

MVP 文档开篇直接给出立场：

> "分析师不再埋头改 agent 的产出，而是用一句自然语言指令让 agent 去改 ——「承付义务这段太浅，拆出期限分布，重点说近两年占比」。这句指令本身就是最高质量的训练信号：它无损地说清了「哪里不对、该怎么改、为什么」。系统把这些指令攒起来，定期让反思器提炼出反复出现的模式，改写进 skill。下次同类调研，agent 第一次产出就已经包含了这个判断。"

为什么只保留"指令驱动"这一种修正通道，而不要"直接改产出"那种？文档解释得很硬：

> "「直接改产出」只留下 diff，「为什么」丢了，得靠 agent 反推 —— 有损、可能推错。「指令驱动」里，「为什么」是人亲口说的 —— 无损、零推断，还省掉整个归因环节。"

底层有一个很简单的纪律要求：**改东西必须"说出来"，不能埋头自己改。** 这恰好和 Bttr 关于 Company Context 的那句"If it was not recorded, it did not happen to the intelligence"完全咬合 —— 没说出来的判断，对 skill 而言就是没有发生过。

---

### 2.1 动作 ① · 记录指令

**做什么**：分析师在和 agent 的对话里，本来就会说"这段改成……""补一下……"。MVP 只需在 agent 收到这类修改指令时，顺手把指令记一条。

**怎么做**：agent 每次执行完一条修改指令，自己调一个 `log_instruction()` 写库。文档给的伪代码：

```python
def on_revision_instruction(user_msg, ctx):
    log_instruction({
        "task_type":           ctx.task_type,           # prospectus_research
        "skill_id":            ctx.skill_id,
        "skill_version":       ctx.skill_version,       # v3
        "target_section":      ctx.current_section,     # 承付义务分析
        "instruction":         user_msg,                # 人的原话，一字不改
        "agent_output_before": ctx.last_output_of_section,
    })
    # 然后正常执行修改…
```

**唯一的判断**：什么算"修改指令"？文档原文："不是所有对话都要记。只记「针对已有产出、要求改进」的指令。简单规则：agent 自判 ——「这条消息是在让我修改刚才的产出吗？」是 → 记录；否（新任务、闲聊、提问）→ 跳过。MVP 可以先宽松记录，后续用 issue_category 过滤噪声。**宁可多记，不可漏记。**"

**这一步在五层循环里的位置**：Sensor 层。它在补传感器 —— 把原本只在分析师嘴里飘走的修正信号，落到一个 agent 看得见的地方。

---

### 2.2 动作 ② · 入库

**做什么**：把指令事件写进 SQLite 的 `instructions` 表。MVP 阶段就这一张主表，9 个字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | INTEGER PK | 事件唯一 ID |
| `ts` | TIMESTAMP | 发生时间 |
| `task_type` | TEXT | 任务类型，如 prospectus_research |
| `skill_id / skill_version` | TEXT | 本次用的 skill 及版本 |
| `target_section` | TEXT | 指令针对的章节，如"承付义务分析" |
| **`instruction`** | TEXT | **【核心】人说的原话，一字不改。这就是一手的 why** |
| `issue_category` | TEXT | 问题归类（可选，反思器可自动打） |
| `agent_output_before` | TEXT | 被指令针对的那段 agent 原产出（给反思器上下文） |
| `promoted_to_skill` | BOOLEAN | 是否已被进化进 skill（防重复消费） |

文档强调的"砍掉了哪些字段"很关键：

> "✗ why_reason —— 被 instruction 取代（指令本身就是 why）
> ✗ revised_output —— MVP 不必存改后稿，指令已含意图
> ✗ reason_source / severity / embedding —— 归因相关，全部不需要
> 一张表从 13 个字段瘦到 9 个，且没有一个需要「推断」。"

为什么强调"没有一个字段需要推断"？因为推断会引入噪声 —— 一旦你让系统去猜"分析师为什么改"，就会有猜错的时候，错误一旦进库就污染后面的反思器。**指令本身就是 why，无需推断**，是这套设计最干净的地方。

配套表 `skill_versions` 让 skill 可回滚：每次进化留痕，新版必须 ≥ 基线分才能上线，否则丢弃。这是"不退步"的硬约束，MVP 也不能省。

**这一步在五层循环里的位置**：Company Context 中心。每一条入库的指令，都是公司大脑里多了一条可读的判断。

---

### 2.3 动作 ③ · 反思器提炼

**做什么**：定期或攒够触发，把同类指令喂给反思器，归纳出反复出现的模式，产出"skill 补丁建议"。

**触发时机**：

- 按量：某 task_type 下未消费的指令攒够 N 条（建议 N=5–8）
- 按期：每周固定跑一次

**反思器 prompt**（文档原文，可直接用）：

```
你是 skill 的「进化器」。下面是 skill 当前版本，以及最近一批
分析师在使用它时下达的【修改指令】(这些指令直接代表了
人类的判断:哪里不够好、应该怎么改)。

【当前 skill v{n}】
{skill_content}

【最近的修改指令】
{instructions: [{instruction, target_section, 
                 agent_output_before}, ...]}

请你:
1. 找出【反复出现 ≥2 次】的指令模式,忽略一次性的特例
2. 为每个模式,提出对 skill 的最小必要修改(增量,别重构)
3. 指明改动放在 skill 哪一节、新增或修改哪条规则
4. 标注每条改动消费了哪些 instruction id(供追溯)

输出 JSON:
{ patches: [{section, change_type, new_rule, 
             based_on_ids}], rationale }
```

注意"反复出现 ≥2 次"和"忽略一次性的特例"两条 —— 这是反思器的"信号去噪规则"。一次性的指令可能是分析师当下情境下的特殊需求，硬塞进 skill 反而会让所有未来任务都被它绑架；只有反复出现的模式，才说明这是一个**普遍判断**，值得变成 skill 的一部分。

**这一步在五层循环里的位置**：Learning Mechanism。它就是 Bttr 那句"By morning it knows more than it did last night"的工程化实现 —— 只不过被改进的不是代码，而是 skill 的判断条款。

---

### 2.4 动作 ④ · 验证后进化

**做什么**：反思器吐出补丁建议之后，不直接上线，要先过两道闸 —— 留出集自动验证 + 人工最终确认。

**MVP 文档原文**：

> "**留出集**。攒一组人改到满意的「黄金终稿」做留出集。skill 改写后在上面重跑，LLM-as-judge 打分。
> **只升不降**。新版得分 ≥ 基线才进入候选；否则丢弃（但保留记录，失败也是数据）。
> **人工上线**。通过验证的候选，由你一键确认上线。skill 变更是高风险动作，人在闭环。"

这两道闸特别关键，因为**自我改写本身有退步风险**。文档直接点了一句业界铁律：

> "自我改进只在结果可验证处有效。"

也就是说，让一个系统改自己是危险的 —— 除非你有一把"客观尺子"在那里量它改完之后是不是真的变好了。留出集就是那把尺子。

**这一步在五层循环里的位置**：Quality Gate。它把"循环自我改进"这个高风险动作的边界，画在了"一键人工确认"这个具体的节点上 —— 完全对应 Bttr "the line between autonomy and oversight is set here, not at the prompt"。

---

### 2.5 简化图示

```
分析师说:「承付义务太浅,拆期限分布,重点近两年」
        ──①记录──▶  捕获指令原文 + 它针对哪个章节/产出
        ──②入库──▶  instructions 表（instruction 字段 = 一手 why）
              │
              ▼ （攒够 N 条同类 / 每周触发）
        ──③提炼──▶  反思器读同类指令 ──▶ 归纳「重复模式」
        ──④进化──▶  改写 skill ──▶ 留出集验证 ──▶ ✓人工上线
              │
              ▼
        skill v(n+1) ── 下次 agent 第一次产出就带上这个判断
```

文档对这张图的总结：

> "对比上一版方案：没有归因 agent、没有 diff 捕获、没有混合确认、没有 severity 推断。指令是一手信号，直接用。"

## 第 3 章 · 两者咬合 · 组织级飞轮

把第 1 章和第 2 章合并起来看，会发现一件让人愉悦的事 —— **指令驱动 Skill 进化引擎不是"五层循环"之外的另一个东西，它就是五层循环在一个具体业务（投研调研）上的实例化**。每一个动作都对应着 Bttr 五层中的一层。

### 3.1 五层映射表

| Bttr 五层（+ 中心） | Skill 进化引擎里的对应物 | 在 SpaceX 案例里具体表现 |
|---|---|---|
| Sensor 层 | 动作 ① 记录指令 | 分析师那句"承付义务太浅"被 `log_instruction()` 接住 |
| Policy 层 | "什么算修改指令"的判断规则 + skill 谁能改的权限 | 只记针对已有产出的修改指令；skill 改动必须经反思器 + 人工 |
| Tool 层 | `log_instruction()`、SQLite 写入、反思器 prompt 调用、留出集打分调用 | 一组确定性 API，agent 不靠魔法做事 |
| Quality Gate | 留出集 + LLM-as-judge + 人工一键确认 | v3→v4 候选打分 0.78→0.85；分析师拍板上线 |
| Learning Mechanism | 动作 ③ 反思器 + 动作 ④ 进化 | 一周内 6 条同类指令被归纳为"大额数字必拆结构"规则 |
| Company Context | `instructions` 表 + `skill_versions` 表 + skill 内容本身 | 分析师说过的所有判断，永久可读、可追溯、可回滚 |

这张表的价值不在于映射本身，而在于它告诉你：**"做一个能让 skill 进化的系统"和"做一个 AI Native 组织"，在工程上是同一件事的不同切片。**

---

### 3.2 飞轮：从一句话到组织级智能

这是两者咬合后真正的核心 —— **一个组织级的飞轮**：

```
[ 分析师跑一次 SpaceX 调研 ]
        │ 用 skill v3 拿到产出
        ▼
[ 一句修改指令 ]    ← Sensor 接住
        │ "承付义务太浅，拆期限分布"
        ▼
[ instructions 表多了一行 ]    ← Company Context
        │
        │ 反复几次后……
        ▼
[ 反思器周期性提炼 ]    ← Learning
        │ "大额数字必拆结构" 这条规则浮现
        ▼
[ 留出集 + 人工确认 ]    ← Quality Gate
        │ skill v3 → v4
        ▼
[ skill v4 上线 ]    ← Company Context 写回
        │
        ▼
[ 下一次同类调研，第一次产出就更深 ] ← 循环重启，但起点更高
```

注意箭头不是"开始 → 结束"，而是闭环。每跑完一圈，下一圈的起点都更高 —— 因为：

1. **skill 比上一版多了一条规则。** v4 比 v3 知道得更多。
2. **分析师的注意力被释放。** 上一轮被改的那一段，下一轮不用再改了，分析师可以把同样的注意力投在更深的判断上。

这就是 Bttr 文章里那句"slope vs level"的具体形态 —— **水平上每一次循环都在做事，斜率上每一次循环都在变聪明。** 公司的真正复利不来自任何一次循环本身，而来自循环的斜率。

---

### 3.3 为什么这件事值得投入

回到一开始的灵魂拷问 —— 为什么我应该把自己的工作流改造成这个形态？三句话：

**第一，你嘴里那句"该这样改"，今天一次性使用、明天就消失了。** 它能改掉这一次的产出，但下一次同类问题来的时候，你还得重说一遍。一年下来你说了几百遍同一个意思。这种"一次性使用"的判断，才是真正的浪费 —— 浪费的不是 token，是你 —— 一个本应做更高阶判断的人 —— 的时间。

**第二，你说的话只要被记住，就会以复利方式回报你。** 每一句被存到 `instructions` 表的修正，都是 skill 在未来无数次同类任务中替你重复执行的"判断模板"。一句话被存一次，循环跑一万次，这一句话就替你做了一万次决定。这是非线性的杠杆。

**第三，五层循环 + 进化引擎合起来，就是把你这个"调研员"蒸馏成一个永远在岗的智能体。** Bttr 那句被反复引用的话很合用：

> "Senior people will increasingly act like fund partners. Outcome over output."

你不需要再亲手把每一份调研改完，你只需要让你说过的每一句话都被记住、被沉淀、被复用。剩下的让循环去跑。**而你，坐在循环的边缘，做循环做不了的事 —— 新情境下的判断、和客户的谈判、对一份报告的品味判断、深夜里关于公司方向的犹豫。**

> "The company is a brain in the middle. Humans sit at the edge."
> （公司是中间一颗大脑，人坐在边缘。）

你的工作流，下一步该把哪一段说出来？
