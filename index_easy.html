PART 01

五层循环 · 一颗大脑在中间，人坐在边缘

Bttr 在 Burn Tokens, Not Headcount 中提出：传统层级是低带宽时代的应对机制。 AI Native 的公司被重新定义成「五层循环 + 一个公司语境中心」 —— hover 任意卡片，看它在 SpaceX 案例里如何工作。

LAYER 01

Sensor 感知层

The loop only knows what it can sense.
系统的输入感官 —— 客户邮件、工单、代码 commit、产品遥测、内部对话。决定循环的视野边界。 一件没有被传感器接住的事，对系统来说就是没有发生过。
"循环只知道它能感知到的事物。"
在 SpaceX 案例里 分析师对 agent 说出的「承付义务太浅，拆出期限分布」这一句话，被 log_instruction() 接住，写进 instructions 表 —— 没有它，这句话飘走，循环对这次修正一无所知。
LAYER 02

Policy 策略层

Rules about what the system can do without asking.
自治与监督的护栏。哪些动作系统可以自己拍板，哪些必须升级到人。 这是一道挂在循环外侧的硬约束，不是 prompt 里的软约束。
"自治与监督的边界设在这里，不在 prompt 里。"
在 SpaceX 案例里 agent 自己可以重组段落、加交叉引用，但任何"对 SpaceX 估值的最终判断"必须由分析师给出；skill 改写必须经反思器 + 留出集 + 人工确认，不许 agent 偷偷改。
LAYER 03

Tool 工具层

Deterministic APIs. Boring infrastructure.
循环里所有"动手做事"的能力，被暴露成一组确定性 API。 决策面靠模型推理，工具面靠死板可靠。模型再不聪明，也只能调既定 API 做既定事。
"无聊但可靠的基础设施。"
在 SpaceX 案例里 log_instruction()、SQLite 写入、反思器 prompt 调用、留出集 LLM-judge 打分 —— 都是确定性 API。agent 不靠魔法做事，只在选用哪个工具上动脑子。
LAYER 04

Quality Gate 质量闸门

Safety filters + human review for high-stakes actions.
所有从循环里推出去的产出，进入世界前要过的闸门。安全过滤器（自动）+ 高风险动作人工审核。 闸门让"自治"变得可负担。
"高风险动作，人在闭环。"
在 SpaceX 案例里 skill v4 候选要在留出集（3 篇黄金终稿）上跑完 LLM-judge 打分，得分 0.85 ≥ 基线 0.78 才能进入待上线，最后由分析师一键确认 —— 没有这道闸，自我改写就是危险动作。
LAYER 05

Learning 学习机制

By morning it knows more than last night.
循环的自检 + 自改 + 自部署能力。失败 → 诊断 → 补丁 → 上线，在循环本身的运转中自然发生。 多数公司缺的就是这一层。
"循环跑过夜，第二天早晨整个 codebase 更聪明。"
在 SpaceX 案例里 反思器每周读完 6 条 instructions（id 12/15/19/23/24/27），归纳出一条新规则「大额数字必拆结构」，自动产出 patch 候选 —— Bttr 那段"系统看着自己在哪里失败、写出补丁"在投研场景的具体形态。
CENTER · COMPANY CONTEXT

公司语境 · 大脑在中间

If it was not recorded, it did not happen to the intelligence.
环绕在五层中央的"公司大脑" —— 历史、决策、客户、流程、教训、品味。所有让公司之所以成为这家公司的隐性知识，要被显式地、可读地存在一处，让循环里每个 agent 都能读到。 没说出来的，对系统就是不存在的。
"公司是中间一颗大脑，人坐在边缘。"
在 SpaceX 案例里 instructions 表（一手 why）+ skill_versions 表（可回滚的判断历史）+ skill 文件本身。分析师说过的所有判断，永久可读、可追溯、可复用 —— 这就是把一个调研员蒸馏成"永远在岗的智能体"的底层载体。

PART 02

指令驱动 Skill 进化引擎

MVP 设计塌缩到只剩一种修正通道：人用自然语言指令让 agent 改。指令本身就是最高质量、无损的训练信号 —— 哪里不对、该怎么改、为什么，全在一句话里。 下面是一次完整循环：从分析师的一句话，到 SpaceX skill v3 → v4 的真实蒸馏。

第一次产出对比 · 同样的 SpaceX 承付义务段

分析师在第二轮里没有下任何修改指令。v4 第一次写出来的，已经覆盖了上一轮她"说出来"的所有判断。

用 skill v3 写出来的承付义务段（分析师皱眉版）

关于 SpaceX 重大现金承付义务的分析截至 2026-03-31，SpaceX 在 S-1 中披露的重大现金承付义务（Material Cash Commitments）主要包括以下几项： · EchoStar 频谱许可收购：约 196 亿美元总对价，预计 2027-11-30 前后交割。 · 未偿债务：合计未偿本金 291.32 亿美元，最早一笔本金到期日 2027-08-28。 · 最低租赁付款：合计 58.23 亿美元，本财年内到期 10.26 亿美元。 · 第三方云算力多年期容量承诺：S-1 风险因子披露公司有"不可撤销的多年期容量承付义务"。 · 零部件与原材料采购：常态采购协议大多无长期约束订单。 小结综合上述披露，SpaceX 当前承担着规模可观的中长期承付义务，主要由 EchoStar 频谱收购、未偿债务、租赁、云算力承诺等组成。

分析师判断：所有信息都对、所有数字都真，但没用。读起来像把 S-1 翻译了一遍，没有期限分布、没有结构拆分、没有近两年占比、没有"那这又怎样"。

用 skill v4 写出来的承付义务段（分析师没下任何指令）

关于 SpaceX 重大现金承付义务的分析截至 2026-03-31，SpaceX 在 S-1 中披露的重大现金承付义务规模可观，但其现金流压力的真实节奏与披露的总额并不一致。下面按"先拆构成、再排期限、最后看含义"的顺序给出。 一、按构成拆分（区分现金 vs 股本）
类别总额现金部分非现金部分
EchoStar 频谱收购196 亿最高 85 亿（代付 EchoStar 债务）111 亿（261.8M 股 A 类，固定 $42.40/股）
总未偿债务（本金）291.32 亿全部现金（含可变利率）—
最低租赁付款58.23 亿全部现金—
云算力多年期容量未披露全部现金，不可撤销—
关键校正：外界容易把 EchoStar 196 亿当成"近期要现金出去 196 亿"，但其中只有最高 85 亿是现金（且要到 2027-11-30 才支付），111 亿其实是股本稀释。 二、按期限分布（24 个月节奏）· 未来 12 个月（至 2027-03-31）：租赁 10.26 亿 + 利息 + 云算力月付，约 10–15 亿； · 未来 12-24 个月：2027-08-28 起首批债务本金到期 + EchoStar 现金 85 亿（2027-11-30）+ 持续运营，现金集中点； · 24 个月以上：剩余债务本金 + 剩余租赁尾段。 三、so-what · 对现金流的含义1. 未来 18 个月并不构成现金压力高峰：本财年内承付义务实际现金流出主要来自租赁与利息，规模相对可控，不需要为承付义务额外融资。 2. 真正的现金压力点集中在 2027 Q3-Q4：85 亿 EchoStar 现金 + 首批债务本金 + 运营现金流出在同一 6 个月内集中。SpaceX 的 IPO 时点窗口大概率被这个节点倒逼。 3. 111 亿股本稀释的隐性成本：以 $42.40 锁价对 EchoStar 发行 261.8M 股 —— IPO 后股价 vs $42.40 的偏离，决定这笔交易是折价让利还是高估自己。这是估值锚之一。 4. 云算力承诺值得单列警惕：S-1 风险因子明确"non-cancellable, multi-year"。具体数额未公开 —— 同时收购 xAI、运营 Starlink、自建数据中心，这部分可能是隐性现金黑洞。建议追问管理层。

分析师判断：这次没有再下任何修改指令。v4 第一次产出已经包含上一轮她说过的所有判断 —— 那一句"承付义务太浅，拆出期限分布"，永久变成了 skill 的拷问清单第 7 条。

PART 03 · 飞轮

你嘴里那句"该这样改"，
今天一次性使用、明天就消失了。

但只要被记住，它就会以复利方式回报你 —— 一次说出口，未来无数次同类任务自动按这条判断执行。 你坐在循环边缘，做循环做不了的事：新情境下的判断、和客户的谈判、对一份报告的品味判断、深夜里关于公司方向的犹豫。
